# 収録文書の案内

このディレクトリには、リポジトリ内で繰り返し参照する判断基準を置く。設定ファイルやテンプレートを読むだけでは分かりにくい運用意図、配置判断、文書化方針をここで補う。

## この文書の役割

- トップレベル `AGENTS.md` から参照する恒久文書の一覧を示す。
- 各文書が何の判断に使うものかを短く説明する。
- ここに追加してよい文書と、別の場所に置くべき文書を切り分ける。

## 配置分類

`docs/` に置く文書は、次のいずれかに分類する。

- 全体方針
  - リポジトリ全体で優先する価値判断や基本姿勢をまとめる。
  - トップレベル `AGENTS.md` から参照される文書として置く。
- 記述・運用ルール
  - コンテンツ配置、Markdown、GitHub 運用など、複数カテゴリで繰り返し使う実務上の判断基準をまとめる。
  - 設定ファイルの値そのものではなく、設定だけでは分かりにくい背景や使い分けを説明する。
- 収録案内
  - `docs/` 配下の文書一覧、追加基準、他の置き場所との境界を示す。
  - この文書が担当する。

上記に分類できない内容は、原則として `docs/` に置かない。

## docs/ 以外に置く内容

- 公開先へ投稿する本文は、対象カテゴリ配下の記事本文ファイルに置く。
- 対象記事だけの事情や単発の調査メモは、記事別の `*.AGENTS.md` や作業メモに置く。
- 特定のトピックだけで使う執筆・編集ルールは、そのトピック直下の `AGENTS.md` に置く。
- 再利用する作業手順は、`docs/` ではなく `skills/` に置く。
- 整形・lint・CI の設定値一覧は、対応する設定ファイルに置く。

## AGENTS.md との責務境界

- トピック別の `AGENTS.md` は、そのディレクトリ配下で作業するエージェントが最初に読む入口として扱う。
- トピック直下の `AGENTS.md` には、対象ディレクトリの適用範囲、そのトピック固有の執筆・編集ルール、その階層で追加になる短い実務上の注意を書く。
- 下位 `AGENTS.md` には、その階層で追加になる判断だけを書く。
- 下位ディレクトリごとに追加判断があるかを確認してから、`AGENTS.md` の配置を決める。
- 追加ルールがある階層に `AGENTS.md` を置く。
- 同じ判断基準を複数階層で使うときは、最上位の `AGENTS.md` かカテゴリ横断 docs にまとめる。
- 上位 `AGENTS.md` が参照している docs を確認し、下位 `AGENTS.md` には追加判断だけを書く。
- `docs/` の新規文書は、複数トピックで再利用するかを確認してから追加する。
- トピック固有ルールは、そのトピック直下の `AGENTS.md` に書く。
- 公開本文の説明とエージェント向けの作業メモは、置き場所を分けて管理する。

## 命名

- 文書名は、扱う範囲と用途が分かる `kebab-case` にする。
- 全体方針は `repository-philosophy.md`、`content-structure.md` のように主題名で付ける。
- 文書数が増えて一覧性が落ちるまでは、下位ディレクトリを作らず `docs/` 直下に置く。
- 下位ディレクトリ化は、同種の文書が増えてから用途と境界を決めて行う。

## 収録中の文書

- `repository-philosophy.md`
  - このリポジトリで何を優先し、どの観点で判断するかをまとめる。
- `content-structure.md`
  - どのトップレベルディレクトリに何を置くか、迷いやすい配置判断をまとめる。
- `markdown-writing-rules.md`
  - Markdown の書き方、媒体差、本文ファイルと作業メモの分離方針をまとめる。
- `github-workflow.md`
  - Issue、PR、Milestone、ブランチ、コミットに関するリポジトリ共通の運用をまとめる。
