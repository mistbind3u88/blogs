# stories-review-critique-counter-workflow

三題噺などの短編に対して、`レビュー`・`批評`・`逆批評`・`総論`を1ファイルに統合して出力するための手順。

## 基本方針

- 四点セットは分割せず、必ず1つの Markdown ファイルにまとめる。
- 生成プロセスは分割実行方式を採用し、レビュー 批評 逆批評 はそれぞれ別サブエージェント（相当）として独立に作成する。
- 総論は第4サブエージェント（統合担当）が作成し、先に総論を書かない。
- 推薦文ではなく、根拠付きの評価を書く。
- 良い点だけでなく、弱点と反論可能性を必ず併記する。
- `総論`では3節（レビュー/批評/逆批評）を同格に扱い、勝敗ではなく統合的な見取り図を示す。
- `総論`はメタ説明（3節の説明）で終わらせず、3節の主張を根拠として作品そのものの結論を述べる。
- 書き出し後も Markdown の可読フォーマット（見出し/段落/空行）を維持する。
- 行幅に合わせた機械的な途中改行（ハードラップ）をしない。改行は段落境界でのみ行う。

## 分割実行方式（サブエージェント相当）

- 実行は2レーンで並列可とする。`レビュー` は独立レーン、`批評` -> `逆批評` は依存レーンとして進め、最後に `総論` を統合担当が作成する。
- レビュー 生成時は、他2節（批評/逆批評）の下書きを参照しない。
- 批評 生成時は、レビュー の文体や語尾を模倣しない。問題提起の軸を独自に立てる。
- 逆批評 生成時は、批評 の論点に応答してよいが、文面の言い換えで終わらせず独自結論を置く。
- 総論担当は `レビュー` `批評` `逆批評` の3節を一次入力として参照し、作品評価を統合する。
- 4節を同一トーンで均質化しないため、各節に担当ロール（例: 伴走型レビュアー/厳格批評家/擁護的批評家/編集統合者）を明示してから書く。
- 総論担当は原則として元作品本文を直接入力にしない。3節で不足点確認が必要な場合のみ参照し、新規論点の追加はしない。

## 独立視点ルール

- 4節は「別の書き手が書いた」前提で、語り口と評価軸を分離する。
- `レビュー`は長所中心、`批評`は問題提起中心、`逆批評`は反論中心に明確に役割分担する。
- `逆批評`で `批評` を受ける際も、論旨を再構成して独自に書く（言い換えで終わらせない）。
- 各節の冒頭で「上記」「前節の通り」などの参照語を使わない。
- 各節は単体で読んでも成立するよう、必要な前提を節内で簡潔に再提示する。

## 改行ルール

- 本文段落は原則1段落1行で記述する。
- 文の途中（読点・助詞・語句の間）で改行しない。
- 見出し、箇条書き、段落間は空行で区切る。
- 1文だけを独立強調したい場合を除き、過度な短行分割を避ける。

## 出力先ルール

- 既定パス: `stories/three_subjects/reviews/<story_id>_review-critique-counter.md`
- `<story_id>` は `YY_MM_DD` 形式のファイル名に合わせる（例: `24_02_10`）。

## Markdown 構造テンプレート

```markdown
# <作品名> レビュー・批評・逆批評・総論

## 対象

- ファイル: `stories/three_subjects/<story_id>.md`
- お題: `...`

## レビュー

## 批評

## 逆批評

## 総論
```

固定の小見出し（例: `総評` `強み` など）は必須ではない。
文章の論理構造が明確であれば、各節内の構成は自由に設計してよい。

## 文章量の目安

- ユーザー指定がなければ各セクション 1000字。
- 指定がある場合はその字数に合わせる（例: 各約1000字）。
- `総論`も同等の密度で書き、他3節より短くしすぎない。
- 読みやすさ優先の指定がある場合は、各セクション 500〜800字に圧縮する。

## 圧縮プロセス（500〜800字）

1000字前後で作成済みの四点セットを、可読性重視で 500〜800字へ再圧縮する場合は次を行う。

### 0. 前提

- 圧縮は「新規に文章を足す作業」ではなく、「元文の主要主張を残す編集作業」と定義する。
- 字数目標より主張保持を優先する。主張保持の結果として目安より短くなることは許容する。
- 埋め草目的の一般論・教訓文・同型の締め文を追加しない。

### 1. 主張抽出（節ごと）

各節について、元の1000字版から次の3要素を先に抜き出す。

1. 結論（その節で最終的に言っていること）
2. 根拠A（最重要の具体根拠）
3. 根拠B（必要なら補助根拠。なければ省略可）

### 2. 構造固定（節ごと）

圧縮後の節は原則として次の順で構成する。

1. 結論文（1-2文）
2. 根拠Aの説明（3-5文）
3. 根拠Bまたは留保条件（2-4文）
4. 着地文（1-2文）

補足:

- `レビュー`: 長所の結論 -> 具体根拠 -> 価値の着地
- `批評`: 問題点の結論 -> 症状/原因 -> 改善方向
- `逆批評`: 反論の結論 -> 反証根拠 -> 適用条件
- `総論`: 作品評価の結論 -> 3節の統合 -> 実務的な次手

### 3. 削除優先ルール

先に削る対象を明示する。

1. 同義反復（言い換えだけの段落）
2. 具体性のない抽象評価（「重要」「有効」だけで終わる文）
3. 末尾の教訓句・一般論の締め（どの作品にも使える文）
4. 前節の繰り返し説明

### 4. 追加禁止ルール

圧縮工程では、次を禁止する。

- 字数合わせのための短文追記
- 「この観点を〜」「この方針なら〜」のような汎用締めの反復
- 元文にない新論点の追加

### 5. 品質確認

1. 元文の結論が残っているか
2. 根拠が最低1つは具体として残っているか
3. 各節の役割差（レビュー/批評/逆批評/総論）が保たれているか
4. 文章が自然に終わっているか（字数調整由来の不自然な末尾がないか）
5. 可能な範囲で 500〜800字に収まっているか

## コミット運用（必須）

- 1000字版（初稿）と圧縮版は同一コミットに混ぜない。
- 圧縮版は、元の1000字版コミットとは別コミットで記録する。
- スキル更新（手順改訂）は成果物コミットと分離し、独立したコミットにする。

圧縮時の優先順位:

1. 主張の明確さ
2. 根拠の最低限の具体性
3. 文体の滑らかさ
4. 装飾的表現

## チェックリスト

- 見出し階層が崩れていないか。
- `レビュー` `批評` `逆批評` `総論` の4節がすべて存在するか。
- 断定だけでなく根拠が添えられているか。
- 逆批評が単なる言い換えでなく、批評への実質的反論になっているか。
- 総論が「どれが正しいか」の裁定でなく、3節の有効範囲と併存条件を整理できているか。
- 総論が「各節の性質説明」に留まらず、作品への最終見立て（何が達成で何が課題か）を明示しているか。
- 出力ファイルが段落単位で適切に改行されているか（文途中改行がないか）。
- 編集や追記の後もフォーマットが崩れていないことを最終確認したか。
- 4節が同じ語彙・同じ論理展開に寄りすぎていないか（筆致の差があるか）。
- 圧縮指定時は、各節が 500〜800字に収まり、削減後も根拠が消えていないか。
- 圧縮指定時は、字数調整のための定型補完文を追加していないか。
- 圧縮後の各節が、元の1000字版の主要主張（結論+主要根拠）を保持しているか。
