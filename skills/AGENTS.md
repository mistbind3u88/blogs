# skills/AGENTS.md

## Purpose

- `skills/` 配下で管理する repo 固有 skill の追加・更新ルールを定義する。
- 各 skill は、このリポジトリで繰り返し使う題材固有、運用固有、構成固有の判断を残す場所として扱う。

## Responsibility Boundary

- repo 固有 skill には、このリポジトリでしか決められない判断を優先して書く。
- 汎用的な Git 操作、commit、push、PR 作成、CI 監視、lint 実行、format 実行のような共通開発手順は、原則として skill 本文へ再記述しない。
- 共通手順に依存する場合は、その操作を逐語で書くのではなく、「この skill の責務外」であることを明示し、repo 固有の入力や判断材料だけを書く。
- repo 固有 skill 同士でも責務が重ならないようにし、題材別 workflow、GitHub 文面方針、集計手順などの境界を分ける。

## Independence

- `skills/` 配下だけを読んでも、このリポジトリで何を判断する skill か分かるように書く。
- 外部の個人用 skill 名や実行環境固有の事情を前提にしない。
- ただし、このリポジトリの上位ルールで定義済みの方針は重複展開せず、必要最小限の参照に留める。

## Writing Rules

- `SKILL.md` には、対象、前提、repo 固有の判断基準、出力や成果物の考え方を書く。
- description は「何をする skill か」だけでなく、「どんな依頼で使うか」まで含める。
- 実行コマンド例を置く場合でも、repo 固有の入力値や対象範囲を決めるための例に留める。
- チェックリストには、repo 固有の確認項目だけを書く。format、lint、push 前確認のような共通項目は原則として重ねない。
- 既存 skill を更新するときは、近い役割の skill と責務が重複していないかを先に確認する。

## File Layout

- 1 skill 1 ディレクトリを基本とし、必要なファイルだけを置く。
- `agents/openai.yaml` を持つ場合は、`SKILL.md` の責務と一致する表示名、説明、既定プロンプトにする。
- scripts や assets を追加する場合は、同じ処理を毎回書き直していないか、repo 固有の再利用価値があるかを確認する。

## Review Focus

- 新しい skill を追加するときは、次を確認する。
- この repo 固有の判断が入っているか。
- 汎用開発手順と MECE になっているか。
- 既存の repo 管理 skill と責務が重なっていないか。
- description と本文が、実際のトリガー条件と一致しているか。
