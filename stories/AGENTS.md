# `stories/` 配下で作業するエージェント向けガイド

このファイルは、`stories/` 以下（`stories/three_subjects/` を含む）の作業すべてに適用されます。  
目的は、このディレクトリ配下の文章（短編、三題噺など）の執筆・推敲をサポートしつつ、作者本人の文体やワークフローを壊さないことです。

## 基本方針

- ここにある Markdown は「コード」ではなく「文章」が主役。
- 既に書かれている文体・トーン・見出し構造をできるだけ維持する。
- ユーザーが「手伝って」と言ったときは、
  - 小さくてレビューしやすい変更
  - 具体的な提案を優先し、全面的な書き換えは求められない限り行わない。
- 自動生成した文章を、あたかもユーザー本人が書いたかのように大量に注ぎ込まない。

## ファイルの扱い

- ユーザーの明示的な指示なしに、stories 配下のファイルを移動・改名・削除しない。
- 新しいファイルを作るときは:
  - そのディレクトリで既に使われている命名規則に合わせる。
    - 例: `stories/three_subjects/24_04_11.md`（`YY_MM_DD.md` 形式）
  - 可能なら同じディレクトリのテンプレートや既存の類似ファイルをコピーしてから編集する。
- 既存ファイルを編集するときは:
  - 不要な段落折り返しの変更や空行削除を避ける。
  - 見出しレベル（`#`, `##` など）をむやみに変えない。

## 三題噺 (`stories/three_subjects/`)

- 新しい三題噺ファイルを作成する場合は、原則として `stories/three_subjects/TEMPLATE.md` をベースにする。
  - 「タイトル」「お題リスト」「本文」「メモ・振り返り」など、大きなセクション構造はテンプレート／最近のファイルに揃える。
- タイトル・見出し:
  - 既存の三題噺のファイル（最新のもの数件）を参照し、同じ書式を真似る。
    - 例: `# タイトル｜三題噺 '24_04_11` のような形。
  - 新しい書式を勝手に発明せず、既存のパターンを踏襲する方向で提案する。
- skills との連携:
  - ユーザーが次のスキル名を指定した場合は、その SKILL.md のルールを優先する。
    - `$blogs-issue-pr-workflow`
    - `$stories-three-subjects-workflow`
    - `$stories-three-subjects-finish-workflow`
  - Issue / PR / ブランチ名や、三題噺の説明の書き方は、これらのスキルに合わせる。
  - 作品本文の変更と、リポジトリ設定・メンテナンス系の変更（lint 設定など）は、別コミット・別 PR として扱うよう助言する。

## 文章スタイル

- stories 配下のコンテンツ生成・編集は、特に指定がない限り日本語で行う。
- 既存の文書をよく観察し、その作品ごとの文体・一人称・語り口に合わせる。
- タイトル案・お題の使い方・構成案などを求められたときは:
  - 2〜3 個程度の候補に絞って提案する。
  - ユーザーが望む場合のみ、それぞれの案の狙いや違いを短く説明する。

## 安全とスコープ

- 実在の人物名、現実のセンシティブな出来事、外部サービスや URL などは、ユーザーが明示的に求めた場合のみ物語へ持ち込む。
- テーマやジャンルを大きく変える改稿（例: シリアス → コメディ への全面変更など）は変更範囲が大きいことを会話で一度確認してから提案・適用する。
