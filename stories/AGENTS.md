# 創作・三題噺記事向けガイド

## 基本方針

- ここにある Markdown は「コード」ではなく「文章」が主役。
- 既に書かれている文体、トーン、見出し構造をできるだけ維持する。
- ユーザーが「手伝って」と言ったときは、小さくてレビューしやすい変更と具体的な提案を先に出す。
- 自動生成した文章を提案するときは、ユーザーが選んで取り込める分量に絞る。

## ファイルの扱い

- `stories/` 配下のファイル操作は、ユーザーの指示内容と対象ファイルを確認してから行う。
- 新しいファイルを作るときは、そのディレクトリで既に使われている命名規則に合わせる。
- 可能なら同じディレクトリのテンプレートや既存の類似ファイルをコピーしてから編集する。
- 既存ファイルを編集するときは、本文の意味に関わる変更を優先し、段落折り返しや空行は既存の形を基準にそろえる。
- 見出しレベルを調整するときは、既存ファイルの構造とテンプレートの対応を確認する。

## 三題噺

- 新しい三題噺ファイルを作成する場合は、原則として `stories/three_subjects/TEMPLATE.md` をベースにする。
- 「タイトル」「お題リスト」「本文」「メモ・振り返り」など、大きなセクション構造はテンプレートや最近のファイルに揃える。
- タイトルや見出しは、既存の三題噺のファイル（最新のもの数件）を参照し、同じ書式を真似る。
- 書式を提案するときは、既存のパターンを基準に近い候補から提示する。

## skills との連携

- ユーザーが `$blogs-issue-pr-workflow`、`$stories-three-subjects-workflow`、`$stories-three-subjects-finish-workflow` を指定した場合は、その `SKILL.md` のルールを優先する。
- Issue、PR、ブランチ名や三題噺の説明の書き方は、これらのスキルに合わせる。
- 作品本文の変更と、リポジトリ設定やメンテナンス系の変更は、別コミット、別 PR として扱うよう助言する。

## 文章スタイル

- `stories/` 配下のコンテンツ生成・編集は、特に指定がない限り日本語で行う。
- 既存の文書をよく観察し、その作品ごとの文体、一人称、語り口に合わせる。
- タイトル案、お題の使い方、構成案などを求められたときは、2〜3 個程度の候補に絞って提案する。
- 案の狙いや違いを説明するときは、短く比較できる形で添える。

## 安全とスコープ

- 実在の人物名、現実のセンシティブな出来事、外部サービスや URL を扱うときは、ユーザーの明示要望を確認してから本文へ入れる。
- テーマやジャンルを大きく変える改稿は変更範囲が大きいことを会話で一度確認してから提案・適用する。
